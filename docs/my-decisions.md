## 🎯 컨텍스트 요약 (모든 결정의 기반)

사용자의 답변에서 읽은 핵심 신호:
- **"실질적으로 돈을 벌어야 한다"** — 취미 아니고 실전
- **"성장형"** — 소액 시작, 검증되면 증액
- **매일 2시간 이상** 투입 가능
- **엄격한 테스트** 선호 (Option 1)
- **페이퍼 4주 + Sharpe 1.0 + DSR 0.5** 엄격한 전환 기준
- **Claude Max 20x** 플랜, 다른 작업과 공유
- **전략 우선** (성공이 최우선)

→ 이 결정들의 공통 원칙: **"검증된 안전성 > 빠른 속도"**, **"엄격한 실전 준비"**

## 1. 백테스팅 프레임워크 (A-2)

### 결정: **nautilus_trader (메인) + vectorbt (리서치 보조)**

**근거:**
- 사용자가 "실질적으로 돈을 벌어야 한다" + "엄격한 테스트" + "전략 우선"을 명시
- Freqtrade는 2주 내 MVP 가능하지만 **체결 현실성을 타협** — 실전 괴리 위험
- nautilus_trader는 학습 곡선이 2~3주로 길지만:
  - **백테스트/페이퍼/라이브가 동일 코드** → 가장 흔한 실패 원인인 "backtest-live gap" 방지
  - Rust 코어로 속도와 정확성 모두
  - 실전 체결 시뮬레이션(L2 호가창, 슬리피지, 부분 체결)이 가장 엄밀
- 단, nautilus_trader는 파라미터 스윕(많은 조합 빠르게 테스트)에 약함
- → **vectorbt**를 리서치 단계에서 파라미터 스크리닝 용도로 사용 (최종 검증 X)

**역할 분담:**
```
vectorbt          →  빠른 파라미터 스윕, 아이디어 검증, 스크리닝
                     "EMA 기간 5~50 중 어느 게 좋나?" 같은 탐색
                     |
                     ↓ (승자 파라미터만 전달)
                     |
nautilus_trader   →  최종 엄밀 백테스트 (DSR, walk-forward, CPCV)
                     페이퍼 트레이딩 (실시간)
                     라이브 트레이딩 (같은 코드)
```

**매일 2시간 × 몇 주 = 학습 가능.** 학습 투자의 ROI가 가장 높음.

**업비트 어댑터**: nautilus_trader는 업비트 네이티브 미지원 → `ccxt` 기반 커스텀 어댑터 작성 필요 (초기 작업량 +3~5일).

## 3. LLM 호출 빈도 및 모델 티어 (B-2)

### 결정: **혼합 티어 + 중간 빈도**

**Claude Max 20x 플랜 특성 고려:**
- 토큰당 과금이 아니라 **rate limit 기반** (5시간 창당 메시지 수 제한)
- 다른 작업(일반 Claude 사용)과 공유 → 봇이 과도하게 점유하면 다른 작업 방해
- Max 20x는 전형적으로 5시간당 수백 메시지 → 봇에 할당 가능 여유 충분

### 권장 호출 스케줄

| 빈도 | 작업 | 모델 | 일일 호출 수 |
|------|------|------|:-----------:|
| 매 4시간 | Reddit 감성 점수 + 레짐 내러티브 (한 호출로 통합) | **Sonnet** | 6 |
| 매일 1회 | 일일 성과 요약 + 다음날 주의 사항 | **Sonnet** | 1 |
| 매주 일요일 | 코인 화이트리스트 재선정 + 주간 전략 리뷰 | **Opus** | 1/주 |
| 매월 1일 | 월간 심층 분석 (전략 파라미터 튜닝 제안) | **Opus** | 1/월 |
| 이벤트 기반 | 서킷 브레이커 발동 시 상황 요약 | **Sonnet** | 이벤트당 1 |

**합계:**
- **일일**: ~7~8 호출 (Sonnet 위주)
- **주간 추가**: Opus 1회
- **월간 추가**: Opus 1회

→ **일반 작업 여유 충분히 확보**. 피크 시간 피해서 스케줄링 (예: 새벽 시간대 선호).

### 왜 Opus를 매일 안 쓰는가
- 매 4시간마다 Opus 돌리면 Max 20x 한도의 상당 부분 소진 → 다른 작업 방해
- **감성 압축** 같은 일상 작업은 Sonnet이 품질 차이 거의 없음
- Opus는 "진짜 검증/심층 분석" 같은 **고가치 희소 작업**에만 집중
- 사용자 원칙 재인용: *"진짜 검증하고 분석하고 결정 등 이런건 비싼 모델"*

### 왜 Haiku를 안 쓰는가
- 원래 비용 때문에 권장했지만 Max 20x는 비용 무관
- Sonnet이 감성 분석에서 Haiku보다 확실히 더 정확
- 일일 호출 수가 적어 rate limit 문제 없음

**승인 요청:** 이 스케줄로 가도 될까요? 호출 수가 너무 많다면 조정 가능.

## 4. LLM 거부권(Veto) 강도 (B-3)

### 결정: **조건부 하이브리드 (다층)**

| 조건 | 강도 | 이유 |
|------|------|------|
| **Reddit 감성이 역사적 하위 5% 극단값** | **HARD** (신규 진입 완전 차단) | 극단 공포는 실제로 후행 수익률과 상관 높음 (Reddit 감성 논문) |
| 감성이 하위 20% (일반적 부정적) | SOFT (포지션 사이즈 50% 축소) | 잡음일 가능성 있음, 완전 차단은 과잉 |
| LLM이 레짐을 "high volatility / crash"로 내러티브 | HARD | 룰 기반 ATR% 상위 10% 필터와 AND 조건 |
| Opus 주간 리뷰가 "이번 주 시장 이상 조짐" 경고 | SOFT (일주일간 포지션 70%) | 주간 리뷰는 질적 판단, 완전 차단은 과함 |
| 긍정 감성 | **영향 없음** | LLM에게 매수 신호 주지 않음 — 거부권만 행사 |

### 핵심 설계 원칙
- **LLM은 "브레이크 페달"만, "가속 페달" 아님** — 매매 신호 생성은 오직 룰 기반 전략에서
- 긍정 감성으로 포지션 늘리는 것 금지 (over-optimism 방지)
- 모든 veto는 **로그에 기록** + 대시보드에 표시 → 사후 검증 가능
- **2주 단위로 veto 효과 분석** — veto가 실제로 손실을 막았는지 통계
- 만약 veto가 **누적 수익을 해치고 있으면 자동 비활성화** 후 사용자 경고

## 5. 알림 이벤트 분류 (C-2)

### 결정: 사용자 규칙 "전부 디스코드, 긴급만 카톡" 정리

| 이벤트 | 채널 | 긴급도 | 이유 |
|--------|:----:|:------:|------|
| 매수 체결 | 디스코드 | 일반 | 확인용, 대량 발생 가능 |
| 매도 체결 | 디스코드 | 일반 | 확인용 |
| **손절 발동** | **디스코드 + 카카오톡** | **긴급** | 실제 손실 발생, 즉시 인지 필요 |
| 일일 수익 리포트 | 디스코드 | 일반 | 하루 1회 요약 |
| 주간 성과 리포트 | 디스코드 | 일반 | LLM Opus 주간 분석 포함 |
| **서킷 브레이커 발동 (Level 1~3)** | **디스코드 + 카카오톡** | **긴급** | 시장 급변, 즉시 인지 필요 |
| **에러/예외 (치명적)** | **디스코드 + 카카오톡** | **긴급** | 봇 중단 가능성 |
| 에러/예외 (경미) | 디스코드 | 일반 | 자동 복구 가능한 것 |
| 시장 급변 감지 | 디스코드 | 일반 | 서킷 브레이커 전 단계 경고 |
| **API 키 만료/에러** | **디스코드 + 카카오톡** | **긴급** | 거래 중단, 즉시 조치 필요 |
| LLM 리뷰 결과 | 디스코드 | 일반 | 대시보드에도 표시 |
| **Kill Switch 수동 발동** | **디스코드 + 카카오톡** | **긴급** | 확인 용도 |
| 페이퍼 트레이딩 상태 리포트 | 디스코드 | 일반 | 일일 요약 |

### 요약 규칙
> **카카오톡 = "지금 당장 봐야 하는 것"**
> **디스코드 = "확인해둘 가치가 있는 것"**
>
> 카카오톡 발동 조건: 손실/오류/시장 급변/보안 문제

### 스팸 방지
- **동일 이벤트 5분 내 중복 발송 금지** (debounce)
- **심각도 축소**: 연속 에러 발생 시 처음 1건만 긴급, 이후 요약
- 하루 디스코드 메시지 > 50건이면 자동 요약 모드 전환

## 6. 대시보드 수동 매매 권한 범위 (C-4)

### 결정: **전체 권한 (Option 3) + 안전장치**

**허용되는 작업:**

| 작업 | 확인 단계 | 이유 |
|------|:--------:|------|
| 수동 매수 | 1단계 (주문 확인창) | 기본 기능 |
| 수동 매도 (개별) | 1단계 | 기본 기능 |
| 봇 일시정지/재개 | 1단계 | 긴급 조치 필요할 수 있음 |
| 봇 파라미터 조정 (리스크, 손절 등) | **2단계 (비밀번호 재입력)** | 전략 근간 변경 |
| **전 포지션 청산** | **2단계 + 5초 취소 타이머** | 돌이킬 수 없음 |
| **Kill Switch (봇 완전 정지)** | **2단계 + 5초 취소 타이머** | 돌이킬 수 없음 |
| 전략 교체 | **2단계** | 근본 변경 |
| 코인 화이트리스트 수동 변경 | 1단계 | 단순 조정 |

### 안전장치
1. **모든 수동 액션은 감사 로그(audit log)에 기록** — 타임스탬프, IP, 세션 ID
2. **30분 무활동 시 자동 로그아웃**
3. **2단계 확인이 필요한 액션은 비밀번호 재입력** (세션 비밀번호가 아닌 별도 transaction password 권장)
4. **5초 취소 타이머**: 파괴적 액션은 클릭 후 5초 동안 "취소" 버튼 표시
5. **대시보드는 읽기 전용 기본 모드** — "편집 모드" 토글 필요 (실수 방지)
6. **봇 운영 중 파라미터 변경 시 즉시 적용 vs 다음 봉부터 적용 선택** (권장: 다음 봉부터)

### 조회 전용 UI
- 실시간 포지션 현황
- 수익률 (일/주/월/전체)
- 최근 매매 이력 (100건)
- 현재 레짐 상태 + LLM 해설
- 거시경제 지표 대시보드
- 봇 상태 (건강도, 마지막 체결 시간 등)
- 서킷 브레이커 상태

## 7. API 키 보관 방식 (D-1)

### 결정: **macOS Keychain + Docker 환경변수 주입 + 다층 보안**

**계층 구조:**
```
Layer 1: 업비트 API Key 자체의 권한 제한
  - 출금 권한 OFF (거래 권한만)
  - IP 화이트리스트: 로컬 PC IP만 등록
  - 키 만료일 설정 (예: 90일마다 재발급)

Layer 2: 로컬 저장
  - macOS Keychain에 저장 (security 명령 또는 Python keyring 라이브러리)
  - 파일 시스템에 절대 저장 금지 (.env 파일도 금지)
  - Git에 절대 커밋 금지 (.gitignore 체계적 관리)

Layer 3: 런타임 주입
  - 봇 시작 시 Keychain에서 읽어 환경변수로 주입
  - Docker 컨테이너에는 환경변수로만 전달 (볼륨 마운트 금지)
  - 컨테이너 내부에서는 Python os.environ에서 즉시 읽어 메모리에만 보관
  - 로그에 키 노출 금지 (마스킹 처리)

Layer 4: 모니터링
  - 비정상 거래량/빈도 감지 → 즉시 알림
  - 예상치 못한 IP에서 API 호출 시도 감지 (업비트 자체 제공)
  - API 키 에러 발생 시 즉시 봇 중단 + 카카오톡 긴급 알림
```

### 왜 Keychain?
- OS 수준 암호화, 시스템 로그인으로만 접근
- 다른 앱이 접근하려면 사용자 확인 필요
