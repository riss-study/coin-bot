## Delta summary

Added an SL priority pre-check in the per-trade classification loop:

```python
SL_RETURN_TOLERANCE = 0.01  # 1% buffer over SL_PCT=0.08
trade_ret = float(tr.iloc[i]['Return'])
if trade_ret <= -(SL_PCT - SL_RETURN_TOLERANCE):  # i.e. <= -0.07
    sl_stop_or_other += 1
    continue
```

If a trade's realized return is at or below -7%, it is classified as
`sl_stop_or_other` before the mask-based branches run. Purpose: prevent SL ↔
time_exits false-positive collisions when the classifier is reused verbatim in
W1-05 (4h timeframe).

## Checklist (delta-focused)

### (1) Soundness of SL priority logic vs fees + slippage

- SL triggers at `close * (1 - 0.08)`. vectorbt then applies slippage (0.05%)
  and fees (0.05% per leg). Realized trade return floor is therefore roughly
  `-0.08 - 0.001 ≈ -0.081`. Always strictly below the -0.07 cutoff → every
  actual SL trade is caught by the pre-check.
- Upper-tail false-positive risk: a non-SL exit (RSI/time) that happens to
  realize ≤ -7% without SL firing. At daily granularity, if a trade is down ≥ 7%
  at the exit bar close, a -8% SL would almost always have fired first (same-bar
  SL fills in vectorbt trigger on the intrabar low when low crosses the stop).
  Scenario where SL did not fire but close return ≤ -7%: possible only if the
  bar's low never touched the SL level but close did — a ≈1 percentage point
  strip. Acceptable adversarial risk; in practice, any such trade is
  economically equivalent to an SL trip and attributing it as
  `sl_stop_or_other` is actually correct semantics.
- Tolerance of 1% is generous and well-motivated by slippage + fees + intrabar
  noise. Not so wide that it would swallow ordinary RSI/time exits (which have
  bounded losses well above -7% given RSI(4)>50 or 5-bar hold on a mean
  reversion system).

Verdict: sound.
