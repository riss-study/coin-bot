### WARNING (3건)

#### W-1: `pf.trades` vs `pf.positions` 의미 차이 박제 누락

**위치**: 전반

본 미니 테스트는 entry 1회 (bar 50) → 단일 trade. 그러나 vectorbt 0.28.5에서 `pf.trades`는 **Exit Trade** 단위 (부분 청산 분리). `pf.positions`는 Position 단위 (FIFO/LIFO 합쳐진 보유 단위). 본 테스트는 size 100% + 단일 entry/exit이므로 `trades.iloc[0]` = `positions.iloc[0]` 동치이나, 코드 박제에 그 의미를 명시하지 않음. 향후 Strategy C/D Primary grid에서 partial exit 도입 가능성 있다면 trades vs positions 구분 박제 필요.

**완화**: 주석 1줄 추가 — "단일 entry 가정. 부분 청산 시 trades vs positions 분리 검토 필요."

#### W-2: `sl_trail=True` 의미 (entry bar 비율 freeze) 박제 누락 → method A 결과 해석 오류 위험

**위치**: L9 (방법 A 설명) + L107 (method B comment)

vectorbt 0.28.5 `from_signals` docstring 직접 인용 (venv 검증):
```
sl_stop (array_like of float): Stop loss.
    A percentage below/above the acquisition price for long/short position.
sl_trail (array_like of bool): Whether `sl_stop` should be trailing.
```

**핵심 함의**: `sl_trail=True`일 때도 `sl_stop`은 **acquisition price** 기준. entry bar에서 sl_stop_ratio 값을 freeze한 뒤, trailing high에 그 비율을 적용 → trailing_max × (1 - ratio_at_entry).

본 테스트의 manual 시뮬레이션 결과 (venv 직접 실행):
- Entry @ bar 50: ratio = 3*ATR[50]/close[50] = 0.066917 (freeze)
- Exit @ bar 107: trailing_max(close)=199.53 × (1 - 0.066917) = 186.18 → close[107]=186.06 < 186.18 → fire
- vectorbt 산출 exit @ bar 107, exit_price=186.06 ✓ 일치

즉 **방법 A는 사실상 "고정 trailing %" 전략으로 변질**되며 ATR(t) 변동 반영 X. 박제 의도 ("trailing high에서 ATR_MULT × ATR(14) 만큼 떨어진 시점 exit", w2-03 sub-plan L108)와 정확히 일치하지 않음.

**완화**: 주석에 "방법 A는 entry bar의 ratio freeze 후 trailing_max × (1-ratio) 동작. 매 bar ATR(t) 동적 반영 X. 이것이 박제 의도면 OK, 아니면 방법 B 강제 채택" 명시. 사용자 채택 결정 시 이 차이를 명시 보고 책무.

#### W-3: 채택 결정 임계값 0.5% 박제 근거 부재

**위치**: L189
```python
abs(result_a["return_pct"] - result_b["return_pct"]) < 0.005
```

`return_pct`는 fraction (0.005 = 0.5%p of return). 0.5%p 임계값 자체는 합리이나 sub-plan v4 W2-03.1 박제에는 명시 없음. 본 venv 실행 결과:
- A return = 0.2351 (23.51%)
- B return = 0.2652 (26.52%)
- 차이 = 0.0301 (3.01%p) → 0.005 초과 → 방법 B 채택 권고

이 임계값으로는 두 방법이 일치한다고 결론낼 가능성 거의 없음 (방법 A가 ATR 변동 반영 X로 구조적 차이). 0.5% (= 0.005 fraction)이 의도인지, 0.5%p (= 0.5%)가 의도인지 모호. fraction 0.005는 50bps로 매우 엄격. sub-plan에 명시 박제 권고.

**완화**: 임계값을 sub-plan v5에 박제하거나 (예: "exit_idx 일치 + |return_A - return_B| < 0.5%p"), 채택 결정 권고에 "임계값 0.5%p로 설정한 이유"를 코드 주석에 박제.


high와 low를 독립 샘플. 이론적으로 high < low 발생 가능성은 0이나 (둘 다 abs 적용 + close 기준 ±), high < close 또는 close < low 같은 일관성은 보장. 실용적으로 OK이나 robust 코드라면 `high = np.maximum(close, ...)` 형태가 안전. 본 테스트 한정으로는 문제 없음.

### Manual trailing logic

```python
in_position = False
trailing_high = -np.inf
for i in range(len(df)):
    if entries.iloc[i] and not in_position:
        in_position = True
        trailing_high = df["close"].iloc[i]
        continue
    if in_position:
        trailing_high = max(trailing_high, df["close"].iloc[i])
        stop_level = trailing_high - ATR_MULT * atr.iloc[i]
        if df["close"].iloc[i] < stop_level:
            exit_mask.iloc[i] = True
            in_position = False
```

| 항목 | 검증 |
|---|---|
| `trailing_high` 초기값 `-np.inf` + entry bar에서 `close[entry]` 설정 | OK (entry bar에서 trailing_high 정의 보장) |
| entry bar `continue` (당일 exit 평가 skip) | OK (entry/exit same bar 회피, vectorbt 일반 패턴 일치) |
| `trailing_high = max(trailing_high, close[i])` | OK (close 기반 trailing) |
| `stop_level = trailing_high - ATR_MULT * atr[i]` | OK (매 bar 동적 ATR 적용 박제 의도 일치) |
| `if close[i] < stop_level: exit_mask[i] = True` | OK (close 기반 fire) |
| `in_position = False` after fire | OK (재진입 차단) |

**venv 검증**: bar 105에서 fire (close=190.60, trailing_max=199.53, 3*ATR(105)=8.13, stop=191.40, close < stop). exit_price=190.60, return=0.2652.

## 외부 감사관 의견

본 미니 테스트의 **설계 의도와 비교 로직은 정확**하며, 사용자 채택 결정 (방법 A vs B)에 필요한 정보를 산출할 수 있는 구조입니다. 그러나:

1. **B-1 (KeyError)는 치명적 실행 차단 결함**입니다. `records_readable` 컬럼명이 venv 직접 검증 결과와 전혀 일치하지 않으며, 코드는 첫 호출 (test_method_a) 직후 즉시 KeyError로 종료됩니다. handover #16 패턴 재발이며 CLAUDE.md "vectorbt API 추측 사용 금지" 위반에 해당합니다.

2. **W-2 (sl_trail 의미 명시 누락)**는 사용자 채택 결정 시 오해 위험이 있습니다. vectorbt 0.28.5 sl_trail=True는 **entry bar의 sl_stop ratio를 freeze 후 trailing_max에 적용**하는 동작이며, 본 미니 테스트가 검증한 것은 정확히 그것입니다. 박제 의도 "매 bar 동적 ATR(14) 반영"이라면 방법 A는 본질적으로 부적합 → 미니 테스트 결과와 무관하게 방법 B 채택이 정답일 가능성이 높습니다. 본 감사관 venv 직접 실행 결과 차이 3.01%p는 그것을 뒷받침합니다.

3. **W-3 (임계값 0.5%p 박제 근거)**: 임계값 자체는 합리이나 sub-plan v4 박제 부재. 사용자에게 "본 임계값 0.5%p로 설정 = 어떤 가정 기반"인지 보고 책무가 있습니다.

4. **N-1 (JSON 저장 누락)**: W2-03.6 NIT-3 6단 evidence 책무 충족을 위해 결과 JSON 저장 추가 권고. handover #1 trace 보존 패턴 일관.
